# 봇을 2시간 일찍 트리거하고 스크립트가 자정까지 대기한다

## 맥락

'그해오늘' 봇은 KST 자정 게시가 목표다. cron을 자정 직전에 걸었더니 **실제 게시가 새벽 1시를 넘겼다.** 원인은 GitHub Actions `schedule` 트리거의 디스패치 지연으로, 이 저장소에서 상습적으로 60~90분 밀리는 것이 실측됐다.

## 결정

**cron을 `50 12 * * *` (UTC) = KST 21:50로 앞당기고, 스크립트가 `msUntilKstMidnight()`만큼 대기했다가 게시한다.**

가드 두 개를 함께 넣는다.

- `GITHUB_EVENT_NAME === 'schedule'`일 때만 대기. 수동 실행·로컬은 즉시 진행
- 대기값이 `MAX_WAIT_MS`(3시간)를 넘으면 이미 자정을 지나 시작한 것이므로 즉시 게시

## 기각한 대안

| 대안 | 기각 이유 |
|---|---|
| cron 시각만 앞당기기 | 지연 폭이 들쭉날쭉해 이번엔 자정 **전**에 게시됐다. 시각 튜닝으로는 수렴하지 않는다 |
| 자정 게시 포기, "그날 아무 때나" | 그해오늘은 날짜가 콘텐츠다. 날짜 경계가 어긋나면 의미가 깨진다 |
| 외부 스케줄러(Vercel Cron 등)로 이전 | 봇이 Node 스크립트 + R2 다운로드라 러너가 필요하다. 이전 비용이 문제 대비 크다 |

## 결과

- 게시 시각이 실제로 자정에 고정됐다
- **매일 러너를 약 2시간 점유한다.** Actions 분 소모가 늘었다. 지연 폭이 줄면 트리거를 자정 쪽으로 좁히는 편이 낫다
- 가드 두 개 중 하나라도 빠지면 24시간 가까이 잠들거나 로컬 디버깅이 불가능해진다
